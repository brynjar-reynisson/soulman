# Error Report Action Design (Thinking rule + Action `fs-agent`)

**Date:** 2026-07-17
**Status:** Approved
**Phase:** Soulman Phase 2 — first Thinking rule + first `fs-agent` action type

---

## Summary

Defines how Thinking turns a `folder-watcher` Stimulus into a running daily report entry, and how the Action module's `fs-agent` writes that entry to disk. This is a deterministic rule, not open-ended reasoning: any stimulus with `channel == "folder-watcher"` always results in the same `INVOKE_ACTION` decision.

---

## Thinking Rule

**Trigger condition:** `stimulus.channel == "folder-watcher"`.

**Decision:** `INVOKE_ACTION`

```json
{
  "intent": "Log this error to today's daily report",
  "action_hint": "append_daily_report_entry",
  "parameters": {
    "summary": "<one-line LLM-generated summary of the error>",
    "raw_content": "<stimulus.content.raw_text verbatim>",
    "source_path": "<stimulus.channel_metadata.channel_specific.watched_path>/<filename>",
    "occurred_at": "<stimulus.occurred_at>"
  },
  "risk_level": "low",
  "urgency": "normal",
  "expected_outcome": "one entry appended to today's report file",
  "fallback": "if fs-agent fails, retry once; if it fails again, log to episodic memory with error:execution tag and give up silently — a missed report entry is not worth interrupting the human"
}
```

The one-line summary is generated by Thinking during REASON (same LLM session, no extra call) — e.g. "DigitalMe sync failed: connection timeout to remote host." If `raw_content` is empty (the binary-attachment case from `perception-svc`'s design), the summary is written as `"<filename> (binary, see attachment)"` and no raw content follows it.

This is the **only** rule this spec defines. It does not attempt general-purpose error triage, deduplication of similar errors, or severity classification — natural future extensions once this pipeline is proven, not required for v1.

## Action: `append_daily_report_entry` (`fs-agent`)

### Handoff (`thinking-svc` → `action-svc`)

**Correction (2026-07-17, post-implementation):** this section originally showed `Action module.md`'s generic `task_id`/`action_type`/`context`/`expected_output` handoff shape here, while the "Thinking Rule" section above specified `correlation_id`/`action_hint` per `Thinking module.md`'s actual decision vocabulary. Those are two different, incompatible JSON shapes for the same message — an inconsistency in this spec that wasn't caught until `thinking-svc` and `action-svc` were run together end-to-end (each was built correctly against a different half of this document). The wire format is the one from the Thinking Rule section, verbatim — repeated here for clarity:

```json
{
  "correlation_id": "<uuid>",
  "intent": "...",
  "action_hint": "append_daily_report_entry",
  "parameters": { "summary": "...", "raw_content": "...", "source_path": "...", "occurred_at": "..." },
  "risk_level": "low",
  "urgency": "normal",
  "expected_outcome": "...",
  "fallback": "..."
}
```

There is no separate `context`/`expected_output` wrapper — `action-svc` dispatches on `action_hint` directly.

### Behaviour

1. Resolve report directory: `$SOULMAN_ROOT/reports/`. `SOULMAN_ROOT` is an env var pointing at `C:\Users\Lenovo\soulman-dev` or `C:\Users\Lenovo\soulman-prod` depending on deployment — the same explicit env-var-driven dev/prod split `memory-svc` uses for its `SCHEMA` variable, just resolving a filesystem path instead of a Postgres schema name.
2. Resolve filename: `daily-report-<YYYY-MM-DD>.txt`, using the date the error **occurred** (from `occurred_at`), not the date the action runs.
3. Create the file (and `reports/` directory) if either doesn't exist.
4. Append an entry in this format, preceded by exactly one blank line if the file is already non-empty:
   ```
   2026-07-17 14:32  [C:\Users\Lenovo\DigitalMe\errors]  DigitalMe sync failed: connection timeout to remote host.
   <raw_content, verbatim, unindented>
   ```
   Timestamp is `occurred_at` formatted as `YYYY-MM-DD HH:MM` (local time); the bracketed path is `source_path`'s containing directory.
5. Return `{"status": "success", "artifacts": ["<report_file_path>"]}` to the Routing Agent.

### File Format

Plain UTF-8 text, entries separated by exactly one blank line. No headers, no JSON — intentionally simple so it can be read by a human, appended to by a script, or sent verbatim as a notification body (see the daily-report-delivery spec).

## Error Handling

| Failure | Behaviour |
|---|---|
| `reports/` directory doesn't exist | Created automatically |
| Concurrent writes (two errors land near-simultaneously) | The append is a single synchronous open-append-close per invocation; the Routing Agent processes one action at a time (per Thinking's "one active reasoning thread" model), so no explicit file locking is needed this iteration |
| Disk full / write fails | Return `{"status": "failed", "error": "..."}` to the Routing Agent; per the Thinking rule's `fallback`, retried once, then logged and dropped |

---

## Out of Scope (this iteration)

- Deduplicating repeated identical errors within the same day
- Severity-based routing (e.g. critical errors also triggering an immediate notification instead of waiting for the daily digest)
- Report entries for stimuli from any channel other than `folder-watcher`
